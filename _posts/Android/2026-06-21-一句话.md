## Binder系列

binder的内核态其实并没有新的进程/线程用来专门处理binder传输过程中的内存分配。其原理是复用的client/server用户态的内核栈来执行。

1. 其实用户态到内核态的转变并没有切换线程/进程，因为内核态的地址索引是全局共享的，不是类似用户空间的地址范围，也不是以进程/线程进行切割访问权限控制的，陷入内核态就是进入另一个空间。虽然用户态和内核态都是基于MMU进行虚拟地址访问控制的，但是内核态的mmu在kmalloc的时候范围只在内核态，而不是用户态。
2. 用户态进程隔离访问控制是基于CPU页表的。正常一个进程访问/proc/self/maps是虚拟地址，比方说A进程有0xaaaa地址，B进程也有这个地址。那么这两个进程资源访问的切换不重叠的原因就是CPU页表的不同才能让A访问0xaaaa地址的时候访问的不是B进程的0xaaaa地址
3. binder使用/dev/binder设备文件节点，在open("/dev/binder")的时候其实内核逻辑就会区分当前节点是否是设备节点 还是vfs的普通文件节点，正常文件节点是基于红黑树进行寻址的（这里的寻址是基于硬盘的），而设备树节点在注册的时候会注册某个syscall函数对应使用某个设备文件节点的handle。比方说binder.c中，binder设备的mmap,open处理函数就是binder.c中的binder_open函数。：
   const struct file_operations binder_fops = {
	.owner = THIS_MODULE,
	.poll = binder_poll,
	.unlocked_ioctl = binder_ioctl,
	.compat_ioctl = compat_ptr_ioctl,
	.mmap = binder_mmap,
	.open = binder_open,
	.flush = binder_flush,
	.release = binder_release,
};

4. 因为b进程可以通过provider等方式把ibinder对象传给a进程，但是由于虚拟地址mmu的控制 a进程的地址空间不能直接访问b的地址空间，所以需要在传输的时候通过内核态来给a进程分配handle，不直接把ibinder对象放到a进程中。如果a进程想与b进程通信则a进程中的ibinder对象其实就是内核的handle指针，

#### Tips
这里其实思考了下为啥binder/跨进程传递数据都要经过内核态。以及不经过内核态是否可行。 

1. 本质上内核栈是每个进程都有的东西。内核不是一个进程，是一片code和数据。可以认为是一个静态的elf文件，但是平时这个文件不用（这里指进程运行用户态时）。一旦涉及到陷入内核态执行的时候就会通过syscall接口来从用户态到内核态的转变。syscall本身就是用户态->内核态的接口。从用户态到内核态的转变本质上是CPU状态的转换。跨进程传递数据本身就需要经过复制和拷贝。从进程b拷贝到内核再拷贝到进程a完成了一次跨进程数据传递。
2. 从用户态指针数据（字符串，数据指针）陷入内核态执行的时候是需要从用户态拷贝到内核态的，而不是将指针拷贝到内核态进行虚拟地址转换的（这里以printf函数为例，其入参在陷入内核态的时候已经完成了对字符串长度的获取。这里也是由C语言指定的'\0'作为分割字符进行取长度的）
3. 不经过内核态传递数据是否可行？答案是可行的。用共享内存创建一片内存空间，通过open vfs创建文件描述符 通过进程间数据传递把文件描述符传递给另一个进程。当然在跨进程传输数据的时候还是需要陷入内核态来进行数据的安全传递的。然后就可以通过这个唯一的fd+自旋读写来进程非通知式的跨进程数据传递

## 为啥官方给的app备份不好使了
1. 根据BackupEligibilityRules.java中isAppBackupAllowed函数来看，android12以上就已经放弃了adb backup的支持。因为adb backup默认是-noapk的，只打包应用的私有文件，而且这其中私有文件还仅局限在/data/data或/data/user/0下，一旦涉及到apk验签或者其他有权限目录下的文件创建（四处拉屎行为），就没办法做到完整的backup。
2. 目前仍能使用adb backup命令进行备份的需要满足以下三个要求：
   1. packageName == "android" 的特例
   2. Sys/特权app显式声明了android.backup.ALLOW_ADB_BACKUP property 为 true
```shell
   <manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.example.myapp">

    <application ...>
        <property
            android:name="android.backup.ALLOW_ADB_BACKUP"
            android:value="true" />
        ...
    </application>
</manifest>
```
   3. manifest里声明了android:debuggable="true"的应用
3. 目前几乎备份都是被换机功能拿走了 android12以下目前从aosp来看指定backup命令还是能看到一些私有数据的
4. 设想了下 如果存在短期root情况下 为啥不直接指定目标app中的文件呢，还要啥backup这么鸡肋的东西。。。

## Bin - adbd
1. 启动条件设置prop就行，设置完prop本身就能自动启动：
   1. setprop sys.usb.config adb          # 或 mtp,adb
      # init 进程会自行 start adbd
      或者直接：
      setprop ctl.start adbd
   2. 如果local的adb直接用shizuku这个开源项目用来连接localhost的adbd来install，这样就能和lspatch/npatch连接了
   3. 对于内核态kvm，vm命令来说 受selinux权限控制，需要crosvm权限组才能正常创建vm环境，untrusted_app是无法正常创建vm环境的，连vm命令都无法正常获取
   4. 对于app shellcommand来说 能拿到很多信息的只有getprop，然后就是根据rooted目录结构进行扫描，扫描到了就可以rw，如果没有那就不好使

## Bin - aflags
1.  com.android.adbd.flags.adbd_restrict_vsock_local_cid 这个flag在android16中新增的，其解决的问题是之前adbd对于vsock（vm/kvm创建的环境与adbd通信的socket接口）是豁免鉴权的。也就是说如果使用vm fake cid=2的情况下与adbd通信是可以直接执行shell命令的，但是这个vm本身是需要使用shell/system权限创建环境的，app无法直接使用kvm内核特性/设备树创建自己的vm空间的

## Bin - bcc
1. 从android11（API 31）开始就逐步deprecated的并行计算框架，其至今未完全移除的目的是为了兼容老版本。因为目前gpu渲染主要是使用vulkan和opengl（目前vulkan无论是移动端还是pc端几乎所有的厂商都会对其进行适配，而bcc框架则是通用兼容层，不是针对每一款vendor的gpu指令做的适配，所以其效能要远低于vulkan和opengl）

## Bin - bootstat
1. 这个工具的主要功能是查看开机时长，距离上次开机的时长，距离上次恢复出厂设置的时长
   1. --set_system_boot_reason	从 kernel cmdline 取出 boot reason 写入 property
      --record_boot_complete	记录"系统启动完毕"时间戳
      --record_boot_reason	记录启动原因（reboot/hard reset/thermal 等）
      --record_time_since_factory_reset	计算距上次恢复出厂的天数
      -l	记录 boot time
      -p 打印启动所有信息到terminal

## Bin - bpfloader
1. 这个工具主要功能是init之后load 内核bpf模块。因为gki的推出 vendor不允许直接load .ko，所以bpf本质上是一个官方的内核hook框架。那既然有hook了 那ebpf用来看看https的明文内容其实也是比较合适的
2. kernelsu本质上还是在原有的内核模块上旁载新的内核模块。syscall，vfs之类的。所以用的不是bpf。
3. bpf的权限没有.ko权限大，为啥推出gki，其实本身就是因为没推出gki的时候vendor每次添加内核模块都要rebuild一遍内核，一崩全崩，所以其实推出gki本身就是从linux宏内核向微内核演进的一个中间态

## Bin - bugreport/bugreportz
1. bugreport目前所有的逻辑已经deprecated了，所有的逻辑都放到了bugreportz里了。
2. bugreportz其主要功能是通过socket连接dumpstatz service来获取日志。其所有的逻辑都在dumpstatz里，bugreportz只是一个壳用于在shell执行。

## Bin - camerserver
1. 与app_process不同，camerserver的主要目的就是把camer注册到serviceManager里等待调用，本身就有。所以这个bin属于启动类型 而不是额外执行的工具类

## Bin - charger
1. 这个命令在recovery下执行，但是在正常开机的情况下是找不到这个可执行文件的。其主要功能是关机充电ui

## Bin - cmd
1. 这个命令是service call的增强版，支持文件描述符和异步等待，使用的是IBinder::shellCommand函数，所以可以支持文件描述符和异步

## Bin - service
1. 这个命令同样是shell调用service的方法之一，使用的是transact函数，需要手动指定和构造参数。其service查询使用的是serviceManager，而servicemanager又是使用的特殊fd（其实本质上普通app查询service也是可以使用fd0查询当前系统中已经启动的所有service）

## Bin - cpu-target-features
